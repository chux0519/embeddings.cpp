## Follow-Up Engineering Notes

- Snowflake correctness is now stable for batch, but C++ batch speed is intentionally conservative because multi-item GTE batches run item-by-item.
- A proper GTE block-diagonal attention graph should replace the item-by-item fallback when performance matters.
- For cleaner per-model memory data, run each model benchmark in a fresh process. The current CLI keeps models loaded while progressing through the model list.

## Snowflake Tuning Pass

Implemented a correctness-preserving single-graph batch path for `Snowflake/snowflake-arctic-embed-m-v2.0` / `GteModel`:

- Removed the item-by-item GTE batch fallback.
- Flattened valid tokens are still used, but attention now receives an explicit block-diagonal `[N, N]` mask so tokens from different batch items cannot attend to each other.
- RoPE cache generation now uses per-token positions that reset within each original input instead of global flattened positions.
- GTE attention now uses `ggml_soft_max_ext` to fuse scale + mask + softmax.

## Snowflake Kernel Pass 3

Implemented and tested a hybrid segmented-attention graph:

- QKV projection, output projection, and MLP stay as flattened large matmuls.
- Attention now slices `q/k/v` by original batch item, computes per-item `QK -> softmax -> V`, then concatenates token outputs back into the flattened stream.
- This removes cross-sample attention score computation and removes the explicit `[N, N]` block-mask tensor.

Correctness:

```text

```

Result: fp16 and q8 internal `batch_vs_single` are now exactly `1.0` for all Snowflake cases. q8 remains correct against Python under the existing q8 threshold from the previous full-reference run.

Finding: the segmented attention graph is cleaner and strictly avoids cross-sample attention work, but it does not materially improve this short-text benchmark. On this workload, Snowflake is dominated by QKV/MLP matmul throughput rather than attention score computation.

Additional kernel experiments:

- `ggml_flash_attn_ext` was wired into the segmented attention path. It passed internal batch-vs-single checks but failed Python-reference correctness, so it was reverted.
- q6/q5/q4 quantization was tested. q6/q5 pass correctness but are slower than q8. q4_k reaches `85.4 text/s` at batch 8, but fails correctness with cosine around `0.94-0.96`, so it is not acceptable for the correctness target.
- Preserving `embeddings.word_embeddings.weight` as fp16 for q4_k did not recover correctness, so that policy was not kept.
- `GGML_OPENMP=ON` was tested and did not improve q8 throughput; the final build is back on `GGML_OPENMP=OFF`.

Updated conclusion: to beat Python CPU and TEI at larger Snowflake batches on this host, the next useful target is not attention masking. It is the matmul kernel path for quantized `up_gate_proj`, `down_proj`, `qkv_proj`, and `o_proj`, or a backend that can match PyTorch/oneDNN throughput on these matrix shapes. The current q8 path already wins memory and small-batch latency, but large-batch throughput is matmul-bound.

Kernel-level findings:

- q8 repack should stay off on this host. `GGML_REPACK_Q8_AVX2=1` regressed q8 attention/MLP throughput badly.
- q4/q4_K repack is useful for MLP weights. The final good mixed model repacks `mlp.up_gate_proj.weight` and `mlp.down_proj.weight` as `q4_K_8x8`.
- `ggml_flash_attn_ext` is correct for the segmented Snowflake path after preserving its output layout `[head_dim, heads, tokens]` and reshaping directly to `[hidden, tokens]`.
- With q4_K MLP + q8 attention, `flash_attn_ext` improved batch 8 from `113.50 text/s` to `127.36 text/s` in the long 11-thread run.
- Python CPU, same long-run settings, measured `128.72 text/s` and `1126.4 MB`; the tuned C++ path measured `127.36 text/s` and `521.4 MB`. Shorter 8-iteration runs measured `131.29 text/s`, so the speed comparison is within noise, while memory is consistently less than half of Python CPU and far below TEI.
